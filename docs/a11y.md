# Accessibility checklist (phase 1)

Use this when adding screens or interactive flows.

## Keyboard and focus

- Every interactive control is reachable with **Tab** / **Shift+Tab** and activates with **Enter** or **Space** where appropriate.
- **Icon-only** buttons (close, settings gear, emoji reactions) have **`aria-label`** or visible text.
- Focus styles remain visible (do not remove outlines without a replacement).

## Screen readers and dynamic content

- **Practice** and similar modes: main question/feedback area uses **`aria-live="polite"`** where content updates without full navigation.
- Results and goal completion: use **`role="status"`** for one-off announcements when helpful.

## Color and motion

- Correctness feedback uses **text/icons**, not color alone.
- Prefer **`prefers-reduced-motion`** for large animations on new UI (respect existing patterns in CSS).

## Touch targets

- On mobile, interactive targets should be at least **44×44 CSS px** where feasible (buttons, reaction controls, grid cells).

## QA

- Run **axe DevTools** (or Lighthouse accessibility) on changed routes before merge.
- Spot-check with **VoiceOver** (macOS/iOS) or **NVDA** (Windows) for new flows.
